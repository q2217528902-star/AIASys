# MCP YAML 加载与合并规则

状态: 主线专题
更新时间: 2026-04-04
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于收敛 AIASys 当前单机个人模式下的 MCP 配置文件加载规则。

它回答的是：

- MCP 配置文件应该放在哪里
- 当前任务级 MCP 如何从 yaml 加载
- 系统模板与工作区配置如何合并
- 配置校验失败时如何处理
- 单机个人模式下 token、api key 当前允许如何写

---

## 2. 核心结论

### 2.1 当前任务级 MCP 主入口是 `mcp/mcp.yaml`

当前任务工作区的 MCP 配置主入口应统一为：

```text
workspaces/{user_id}/{workspace_id}/mcp/mcp.yaml
```

### 2.2 系统模板不是当前事实源

系统可以提供默认模板、市场条目和示例配置。

但当前任务真正生效的 MCP 配置，仍以工作区内 `mcp/mcp.yaml` 为准。

### 2.3 当前阶段允许在 yaml 中直接写认证字段

由于当前主线是单机个人模式，允许在 `mcp.yaml` 中直接写：

- `api_key`
- `token`
- `headers`
- `env`

---

## 3. 推荐文件结构

当前推荐目录结构如下：

```text
mcp/
├── mcp.yaml
└── templates/
```

其中：

- `mcp.yaml` 是当前任务实际加载配置
- `templates/` 用于存放系统导入模板、市场样例或用户备份样例

---

## 4. YAML 顶层结构建议

当前建议的顶层结构至少包含：

```yaml
version: 1
servers:
  tavily:
    transport: stdio
    command: npx
    args:
      - -y
      - tavily-mcp
    env:
      TAVILY_API_KEY: tvly-xxx
    enabled: true
```

必要字段建议：

- `version`
- `servers`

单个 server 常见字段建议：

- `transport`
- `command`
- `args`
- `env`
- `headers`
- `enabled`
- `cwd`
- `timeout_sec`

---

## 5. 加载优先级

### 5.1 第一优先级：工作区 `mcp.yaml`

如果当前任务工作区存在 `mcp/mcp.yaml`，优先读取它作为任务实际 MCP 配置。

### 5.2 第二优先级：系统导入模板

如果工作区尚未创建 `mcp.yaml`，系统可以基于：

- MCP 市场模板
- 系统默认模板

为当前任务初始化一份 `mcp.yaml`。

初始化完成后，后续事实源仍然是工作区自己的 `mcp.yaml`。

### 5.3 不推荐的旧优先级

当前不应继续采用以下路径作为任务主事实源：

- 用户级共享 `.mcp/`
- 仓库根目录散落的临时 yaml
- 前端内存拼装出来的 server 列表

---

## 6. 合并规则

当前推荐采用“初始化合并 + 运行时只读工作区事实源”的模式，而不是每次运行都做多层动态深合并。

### 6.1 初始化时可合并

在以下时机允许合并：

- 从系统市场安装一个 MCP 条目到当前任务
- 使用系统模板初始化当前任务 `mcp.yaml`

此时可以把系统模板内容写入工作区 `mcp.yaml`。

### 6.2 运行时不做复杂隐式深合并

运行时加载时，默认直接读取工作区 `mcp.yaml`。

不建议在执行时再做：

- 系统配置深合并
- 用户默认深合并
- 市场模板深合并

### 6.3 覆盖原则

如果工作区已经存在同名 server：

- 以工作区文件中的同名 server 为准
- 系统模板只在显式安装或更新时改写

---

## 7. 校验规则

后端加载 `mcp.yaml` 时，至少应校验：

### 7.1 文件级校验

- YAML 语法合法
- `version` 合法
- `servers` 为对象

### 7.2 server 级校验

- server 名称非空
- `transport` 合法
- 使用 `stdio` 时必须有 `command`
- `args` 必须为数组
- `env` 必须为键值对象

### 7.3 加载级校验

- `command` 是否可执行
- 关键模板字段是否缺失
- timeout 是否为正数

校验失败时：

- 不应直接损坏任务工作区
- 应保留原文件
- 应向前端返回明确错误信息
- runtime 状态可进入 `missing` 或 `refresh_required` 未完成态

---

## 8. 与市场的关系

MCP 市场负责：

- 展示系统提供哪些 MCP 条目
- 展示简介和模板来源
- 触发“安装到当前任务”

MCP 市场不直接等于当前任务正在使用的 MCP 列表。

真正的当前任务事实源，仍然是：

- `mcp/mcp.yaml`

---

## 9. 与运行态的关系

当 `mcp.yaml` 发生变化时：

1. 先保存工作区文件
2. 将 runtime 标记为 `refresh_required`
3. 当前 turn 默认不热更新
4. 下一次执行前重新读取 `mcp.yaml`
5. 记录本轮实际加载的 MCP 配置快照

---

## 10. 当前建议结论

当前主线应坚持以下规则：

1. 工作区 `mcp/mcp.yaml` 是任务级 MCP 主事实源。
2. 市场模板只用于初始化或导入，不是运行时主事实源。
3. 当前阶段允许在 yaml 中直接写认证字段。
4. 运行时优先读取工作区文件，不做复杂隐式深合并。
5. 文件变化统一通过 runtime 软重建生效。
