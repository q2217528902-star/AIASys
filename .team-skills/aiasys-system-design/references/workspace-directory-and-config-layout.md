# 工作区目录与配置文件布局规范

状态: 主线专题
更新时间: 2026-04-04
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于收敛 AIASys 当前任务工作区的物理目录布局和配置文件落点。

它回答的是：

- 一个任务工作区根目录下应该放哪些正式目录
- 哪些目录属于用户可见工作内容，哪些属于系统 sidecar
- Skill、MCP、运行证据、附件和产物分别应该落在哪里
- 后续实现时，哪些路径可以稳定依赖，哪些不应继续随意漂移

本文档属于系统设计约束，优先服务于后端实现、运行时挂载和前端工作区呈现。

---

## 2. 核心原则

### 2.1 工作区根目录是真实事实源

当前任务工作区应尽量让“用户看到的目录”和“系统真正挂载的目录”一致。

不要再维护一层用户不可见的平行副本，然后让运行时偷偷读取另一套路径。

### 2.2 用户工作内容与系统 sidecar 分层

工作区目录至少要区分两类内容：

- 用户工作内容
- 系统 sidecar 与运行证据

用户工作内容应尽量可见、可浏览、可导出。

系统 sidecar 可以隐藏到以 `.` 开头的目录中，但路径仍应稳定、可追溯、可供后端和调试工具使用。

### 2.3 单个任务工作区拥有自己的 Skill 和 MCP 配置

当前主线下，Skill 和 MCP 都优先解释为当前任务工作区拥有的局部配置，而不是用户级共享副本。

因此：

- Skill 包应放在当前工作区内
- MCP yaml 也应优先放在当前工作区内
- runtime 读取的是当前工作区目录，不是用户级隐藏目录

---

## 3. 推荐目录布局

当前推荐单个任务工作区根目录如下：

```text
workspaces/{user_id}/{workspace_id}/
├── attachments/
├── artifacts/
├── env/
├── knowledge/
├── mcp/
├── skills/
├── workspace/
└── .session/
```

如果当前实现还未完全对齐，可以允许渐进迁移，但目标路径应按这套口径收口。

---

## 4. 各目录职责

### 4.1 `workspace/`

用于承接当前任务的用户主工作内容。

建议放置：

- 用户主动创建或导入的文档
- 任务分析过程中生成、且希望长期保留的工作文件
- 需要在右侧工作区中直接浏览的普通文件

### 4.2 `attachments/`

用于承接原始上传附件和外部导入的文件。

建议放置：

- 初始上传文件
- 外部导入的原始资料
- 需要保留原始版本的输入材料

### 4.3 `artifacts/`

用于承接任务产物和可审计输出。

建议放置：

- 导出包
- 图片、图表、报告等生成产物
- 后续特殊渲染协议落地后的文件资产

### 4.4 `skills/`

用于承接当前任务工作区拥有的 Skill 目录包。

目录规则：

- 每个 Skill 为一个目录包
- `SKILL.md` 是入口文件
- 内部目录结构不做硬编码模板限制

### 4.5 `env/`

用于承接当前任务工作区的 Python 环境物料与环境元数据。

当前主线推荐：

```text
env/
├── pyproject.toml
├── uv.lock
├── .python-version
└── .venv/
```

约束如下：

- `uv` 作为工作区默认环境管理器
- 每个任务工作区可拥有自己的 `.venv/`
- 工作区环境的依赖声明、锁文件与 Python 版本都应尽量落在 `env/`
- 不把环境物料继续塞回系统级隐藏目录
- 重型工具链优先通过 `uvx` 或 `uv tool run` 执行，避免污染主运行时

### 4.6 `mcp/`

用于承接当前任务工作区的 MCP 配置文件。

当前推荐至少支持：

```text
mcp/
├── mcp.yaml
└── templates/
```

### 4.7 `knowledge/`

用于承接当前任务关联的知识资源索引或本地知识材料。

### 4.8 `.session/`

用于承接系统 sidecar、状态文件与运行证据。

当前可包含：

- 执行 journal
- 运行态状态文件
- 研究 sidecar
- 导出证据索引
- 配置快照与版本引用

---

## 5. `.session/` 下的建议结构

当前推荐结构如下：

```text
.session/
├── config/
├── executions/
├── exports/
├── history/
├── research/
└── runtime/
```

### 5.1 `config/`

用于记录：

- 当前任务保存的配置版本
- 本轮执行使用的配置快照引用
- 生效关系与待生效关系

### 5.2 `executions/`

用于记录：

- 执行 journal
- 每轮执行的元数据
- 工具调用证据

### 5.3 `runtime/`

用于记录：

- runtime 状态
- 当前环境信息
- refresh required 标记

---

## 6. 前端可见性建议

前端工作区默认应优先向用户展示：

- `workspace/`
- `attachments/`
- `artifacts/`
- `skills/`

默认不直接把 `.session/` 整体暴露为普通文件树主内容。

---

## 7. 当前实现约束

### 7.1 运行时挂载约束

后端运行时挂载应优先直接挂工作区根目录，并从固定子目录读取：

- `skills/`
- `mcp/mcp.yaml`
- `workspace/`
- `env/`

当前推荐执行模型进一步明确为：

- 主执行解释器默认来自工作区 `env/.venv/`
- 若工作区尚未物化 `.venv/`，允许按 `env/pyproject.toml` 与 `uv.lock` 延迟创建
- 不再把 Docker 镜像视为当前主线默认环境物料

### 7.2 迁移约束

如果现有实现仍有旧路径，允许做兼容读取，但必须满足：

1. 新写入优先写到新路径
2. 兼容读取要有退场计划
3. 不继续往旧路径扩新语义

---

## 8. 当前建议结论

当前主线应按以下物理事实收口：

1. 任务工作区根目录是工作内容与任务级配置的真实事实源。
2. Skill 包放在 `skills/`。
3. MCP 配置放在 `mcp/mcp.yaml`。
4. 工作区默认 Python 环境物料放在 `env/`，并优先使用 `uv` 管理。
5. 用户主工作内容优先放在 `workspace/`，输入材料优先放在 `attachments/`，生成产物优先放在 `artifacts/`。
6. 运行证据和系统 sidecar 统一收口到 `.session/`。
