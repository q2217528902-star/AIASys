状态: 主稿
更新时间: 2026-04-05
适用版本: v0.3.9 当前主线

## 1. 文档目的

本文档用于说明 AIASys 当前系统架构的主骨架，包括主对象、信息架构、配置作用域、能力接入模型和执行证据模型。

本文档描述的是当前系统主线，不负责记录所有历史探索方案。历史材料如需追溯，应查 `archive/`。

---

## 2. 当前系统主骨架

AIASys 当前采用以任务工作区为核心的单主骨架设计。

产品主语义是：

- 任务工作区
- 工作区下的对话
- 设置层
- 市场层
- 执行记录与证据层

实现层当前仍大量使用 `session / conversation` 命名，但这属于底层事实源，不改变产品主对象已经收敛为任务工作区。

当前应进一步收敛的目标对象层级是：

- `workspace`
- `conversation`
- `run`

也就是：

- 工作区是一等长期对象
- 对话是工作区内部的记忆分支
- 执行轮次是对话内部的运行证据单元

当前主线已经明确：

- 不再以兼容旧的 session-first 产品模型为目标
- 可以清理本地真实运行产生的旧对话历史与运行数据
- 后续重构直接围绕 `workspace / conversation / run` 展开
- `session_id` 仅视为当前实现层仍在使用的底层运行键，而不是产品主对象

当前默认运行形态收口为单机个人模式：

- 产品侧不再强调登录流程
- 后端可保留单个默认本地用户作为内部兼容骨架
- 现有按 `user_id / session_id` 分目录和分表的实现暂不强制重写
- 当前主线默认只保留本地执行链路，Docker 相关运行资产下沉到 `archive/`

当前进一步建议把工作区默认环境模型收口为：

- 宿主执行链路默认使用本地 Python
- 工作区级 Python 依赖与虚拟环境优先使用 `uv`
- 重依赖、冲突依赖或一次性工具优先通过 `uvx` / `uv tool run` 隔离执行

这里要特别区分：

- `uv` 适合承担工作区环境物料管理
- `uv` 不等于安全隔离边界

也就是说，在当前单机个人模式下，`uv` 很适合作为 Docker 退场后的默认环境方案；但如果后续重新引入更强隔离诉求，仍应单独设计隔离层，而不是把 `uv` 当成容器替身。

## 3. 架构目标

当前架构目标不是追求抽象数量，而是保证以下几点：

1. 一个概念尽量只有一个主事实源。
2. 目录层、默认层、工作层、证据层清楚分层。
3. 工作区、对话、执行轮次三层对象不能再互相混充。
4. 后端先补描述层和作用域模型，再收口执行层。
5. 前端先做信息分层，不把多个作用域塞进一个组件里混讲。

## 4. 信息架构分层

### 4.1 目录层

回答：系统支持什么。

典型对象：

- Skill 市场
- MCP 市场
- 系统提供的能力目录项

### 4.2 默认层

回答：用户长期默认使用什么。

典型对象：

- 用户默认 Agent 配置
- 用户默认模型与偏好
- 我的 MCP
- 后续变量与密钥配置

### 4.3 工作层

回答：当前任务工作区这次怎么跑。

典型对象：

- 任务级 Agent 配置
- 任务级能力覆盖
- 工作区默认执行绑定
- 当前分支运行态覆盖
- 当前工作区下的对话选择与推进路径

### 4.4 证据层

回答：这一轮实际用了什么。

典型对象：

- 执行记录
- 运行环境状态
- 当前执行实际采用的配置快照
- 当前执行实际使用的能力与工具

当前主线建议把“运行环境”进一步拆成三层理解：

- 工作区层保留 `默认执行绑定`
  - 例如默认 `sandbox_mode`、默认 `env_id`
  - 用于回答“新分支默认怎么跑”
- 对话层保留 `当前分支运行覆盖`
  - 用于回答“这条分支临时改成怎么跑”
- 执行轮次层保留 `runtime_summary / execution record`
  - 用于回答“这一轮实际上用了什么”

### 4.5 对象层级约束

当前主线必须把以下三层对象分清：

1. 工作区层
   - 长期任务资产与任务级配置的事实源
2. 对话层
   - 同一工作区下的记忆分支与推进路径
3. 执行轮次层
   - 单次运行的执行记录、工具调用和证据快照

如果一个实现同时让一个 ID 承担三层语义，就属于待拆分的兼容状态，而不是长期目标态。

## 5. 入口架构

### 5.1 任务工作区入口

任务工作区是用户当前工作的主入口，承接：

- 对话
- 同工作区下的对话切换
- 文件与资料
- 任务配置
- 资源与能力快捷入口
- 执行证据与工作区摘要

### 5.2 设置入口

设置入口承接长期默认配置，不应继续承担当前任务即时工作状态。

### 5.3 市场入口

市场入口承接供给侧目录，不应直接等同于当前任务启用状态或用户默认状态。

### 5.4 主界面壳层

当前主界面壳层应优先理解为：

- 左侧：工作区导航与最小系统入口
- 中间：当前工作区中的当前对话工作面
- 右侧：当前任务工作区

其中右侧“当前任务工作区”默认承接：

- 文件与资料
- 任务资源摘要
- 当前任务产物
- 简短运行摘要

执行记录与证据层不应再与右侧工作区做成并列一级常驻主区。

更稳的做法是：

- 从顶部按钮进入执行记录抽屉
- 从消息、任务状态或工作区摘要进入执行证据
- 把完整日志与执行树保留在诊断层，而不是主工作壳层

## 6. 配置作用域模型

当前系统至少区分以下 scope：

- `system`
- `user`
- `task`
- `runtime`

在需要共享模型时，可后续扩展 `team`，但当前不应提前把团队共享偷换成系统级。

当前第一阶段实现建议进一步收简为：

- `system baseline`
- `workspace local`
- `runtime evidence`

其中用户层如暂时保留，也更适合作为内部兼容骨架或预设来源，而不是当前产品要暴露给用户理解的一层继承真相。

当前对象层级与作用域模型的推荐映射是：

- `workspace` 对应工作层主配置
- `conversation` 对应当前记忆与历史
- `run` 对应 runtime evidence

### 6.1 system

表达系统提供什么目录项和系统级事实，不等于强制启用。

### 6.2 user

表达用户的长期默认配置。

### 6.3 task

表达当前任务工作区的覆盖配置。

### 6.4 runtime

表达执行时的运行环境与运行态事实。

## 7. 能力接入架构

AIASys 当前不把 MCP 视为总模型，而把能力接入视为一等概念。

### 7.1 主要适配器类型

- CLI / 本地命令
- 原生 API / SDK
- MCP

### 7.2 当前原则

- `MCP` 是接入方式，不是总抽象
- `MCP 市场` 是协议型目录，不是全能力总目录
- capability registry 应描述能力通过什么适配器接入
- 不是所有外部能力都必须先包成 MCP

### 7.3 当前阶段的 MCP 配置口径

当前系统明确以单机个人使用为前提。

因此 MCP 在这一阶段的设计优先级应是：

- 先可配置
- 先可加载
- 先可运行

而不是先做完整可视化面板。

当前推荐口径是：

- MCP 配置优先使用工作区或仓库内的 `yaml` 文件
- 后端负责把 `yaml` 解析成运行时真正需要的 MCP 配置结构
- 前端面板不是前置条件，最多只需要只读预览和测试入口

当前补充的稳定落点应统一为：

- 工作区目录布局参考 `workspace-directory-and-config-layout.md`
- 当前任务 MCP 主配置入口为 `mcp/mcp.yaml`
- YAML 加载与初始化规则参考 `mcp-yaml-loading-and-merge-rules.md`

当前阶段允许在 MCP 配置文件中直接写入 token、api key 或其他认证字段。

这不是长期团队模式下的最终方案，只是当前单机个人模式下的简化实现口径。未来如果进入多人协作、共享工作区或审计要求更高的阶段，再把它升级为变量与密钥分层模型。

## 8. 任务配置架构

任务配置是当前任务工作区的统一配置入口。

当前目标语义至少包含：

- Agent 配置
- 能力配置
- 运行环境配置

但这只是前端统一入口，不表示后端领域模型要把这些对象揉成一个大对象。

后端仍应保持：

- agent config
- capability registry / capability projection
- runtime substrate / runtime evidence

三者分层。

当前还需要补一条统一语义：

- Skill 包变更
- MCP 配置变更
- Agent 配置变更

都不应默认解释成“当前运行中热更新”。

当前任务配置默认应挂在工作区层，而不是对话层。

也就是说：

- 新建对话默认复用当前工作区配置
- 清空记忆或切换对话不应默认丢失工作区配置
- 如果未来需要对话级实验配置，应作为新增层显式引入，而不是继续隐式复用 `session_id`

更准确的系统行为应是：

1. 工作区配置先保存
2. 当前任务被标记为 `runtime_refresh_required`
3. 当前 turn 结束前不强制打断
4. 下一次执行启动前，系统销毁旧 runtime 并重建同一任务的 runtime
5. 新 runtime 读取最新的工作区 Skill、MCP 与 Agent 配置

这样可以在不丢失任务历史的前提下，实现接近“动态加载”的用户体验，同时保持与底层 Kimi runtime 的真实能力边界一致。

当前前端入口也应继续收口为：

- 任务配置主入口放在当前任务顶部栏
- 全局设置页不再承担当前任务主配置入口
- 任务配置弹窗至少区分基础、Agent、能力、环境、资源五组

运行态行为也应统一按以下规则理解：

- 工作区配置先保存
- 当前 turn 默认不热更新
- 任务可进入 `refresh_required`
- 下一次执行前软重建 runtime

详细规则参考：

- `.agents/skills/aiasys-system-design/references/runtime-refresh-and-rebuild-contract.md`
- `.agents/skills/aiasys-system-design/references/execution-evidence-and-config-snapshot-spec.md`

### 8.1 三层作用域模型

当前系统推荐固定成三层：

1. 系统目录层
   - 负责系统提供的目录项与资源本体
   - 例如 Skill 市场、MCP 市场、数据库连接器本体、知识库本体、知识图谱本体
2. 用户默认层
   - 负责用户长期默认怎么用
   - 例如用户默认 Agent 配置、长期偏好记忆、默认模型偏好、默认 MCP 候选
3. 任务工作层
   - 负责这个任务当前实际启用了什么
   - 例如任务级 Agent 覆盖、任务级 MCP、任务级 Skill、数据库/知识库/图谱挂载

当前不建议引入“全局工作区”作为单独主对象。
全局层只承接目录与资源本体，不承接当前任务现场。

### 8.2 关键对象分层口径

当前推荐如下：

- Agent 配置
  - 用户默认层：默认 Agent 配置
  - 任务工作层：当前任务 Agent 覆盖
- MCP
  - 系统目录层：MCP 市场、系统目录
  - 用户默认层：我的 MCP / 默认候选
  - 任务工作层：当前任务实际启用的 MCP
- Skill
  - 系统目录层：Skill 市场、系统提供的 Skill 包目录
  - 任务工作层：当前任务拥有和启用的 Skill 包
- 偏好记忆
  - 用户默认层：长期协作偏好
  - 任务工作层：任务记忆、任务术语和任务特定约束
- 数据库
  - 系统目录层：数据库连接器本体
  - 任务工作层：数据库挂载关系
  - 默认不自动挂载，必须由当前任务显式选择
- 知识库
  - 系统目录层：知识库本体
  - 任务工作层：知识库挂载关系
  - 默认不自动挂载，必须由当前任务显式选择
- 知识图谱
  - 当前真相：系统级单图谱
  - 目标态：多图谱本体在系统目录层，任务工作层挂载图谱
  - 当前任务默认不自动挂载图谱，进入任务上下文需要显式挂载

补充前端入口约束：

- 资源目录入口默认落在系统目录层或长期资源层
- 当前工作区挂载视图需要显式传入 `workspace_id`
- 不要让左侧入口通过隐式继承 URL query 在“资源本体目录”和“当前工作区挂载视图”之间偷偷切换

### 8.3 知识图谱演进口径

当前知识图谱仍是系统级单实例实现。

因此如果后续要支持任务级知识图谱配置，推荐顺序不是直接把现有图谱改成工作区私有，而是：

1. 先支持多个图谱本体
2. 再给任务工作区增加图谱挂载关系
3. 最后再决定是否支持“创建工作区专属图谱”这一便捷动作

这样更符合“资源本体多实例 + 工作区选择性挂载”的统一模型。

## 9. 设置架构

设置不是单页，而是一层长期默认配置系统。

当前设置层目标包含：

- 用户默认 Agent 配置
- 模型与偏好
- 运行环境入口
- Skill 市场
- MCP 市场
- 后续变量与密钥管理

当前架构要求设置层与任务层分开，不再互相混写。

在 MCP 这一层，当前设置页不是必须先提供“编辑器型面板”。

如果已经存在稳定的配置文件加载器，那么设置层只需要提供：

- 当前使用的 MCP 配置文件位置
- 当前解析出的 MCP 条目预览
- 连通性测试或报错查看入口

而不必先实现完整图形化编辑。

## 10. 市场架构

市场是供给侧目录，不是正在使用的状态层。

当前明确存在：

- Skill 市场
- MCP 市场

其中 Skill 当前应被理解为目录包，而不是单个 Markdown 文档对象。

当前 Skill 的实现建议进一步收口为：

- 运行时系统目录从独立 builtin Skill 目录读取，例如 `apps/backend/builtin_skills/`
- 当前工作区实际拥有的 Skill 包直接存放在 `workspaces/{user_id}/{workspace_id}/skills/`
- 运行时直接挂载当前工作区的 `skills/` 目录
- zip 导入只要求包内存在可识别的 `SKILL.md` 入口文件，不强制固定目录模板
- 入口预览只读取入口 `SKILL.md`，不再把在线编辑器当默认主流程
- `.agents/skills/` 属于 AIASys 协作/开发 skill 仓库，不应再直接作为用户运行时 Skill 市场来源

后续如果引入更多系统能力目录，应继续在目录层扩展，而不是直接并入任务配置。

对于 MCP，当前更推荐的实现载体是：

- 系统或工作区内的 `mcp.yaml`
- 由后端统一加载与校验
- 任务运行时直接读取解析结果

而不是先把 MCP 配置中心做成重型前端管理系统。

## 10.5 工作区与对话的迁移口径

当前主线已收口为：

- 不以“兼容旧 session-first 模型”为目标继续演进
- 不再为了保留旧对话历史而维持额外兼容层
- 允许在进入新一轮重构前清理旧对话历史与旧运行时数据

当前推荐口径是：

1. 产品文案统一成“工作区 / 对话 / 执行轮次”
2. 新一轮前后端改造直接围绕 `workspace_id / conversation_id / run_id` 目标模型展开
3. 对已确认不再需要保留的旧对话历史，不做额外兼容投影
4. 如个别底层实现暂时还保留 `session_id`，也只作为当前代码重构过程中的临时技术事实，而不是要长期维护的兼容承诺

也就是说，当前主线要避免：

- 一边把产品说成工作区下多对话
- 一边继续为了旧 session 历史保留大量兼容接口、兼容命名和兼容状态面

## 11. 运行与热更新语义

当前系统对任务级配置采取 next-run-only 语义。

也就是说：

- 任务级配置允许动态保存
- 当前运行中的 Agent Session 不承诺热更新
- 新保存的配置默认对下一次执行生效

这条规则同时适用于任务级 Agent 配置和任务级能力覆盖。

对于 Skill 和 MCP，需要把这条规则再说得更具体：

- 允许在任务工作区中动态新增、删除或修改 Skill / MCP 配置文件
- 这些变化不会直接注入当前正在运行的 runtime
- 系统通过“软重建当前任务运行态”来应用变更

这里的“软重建”指的是：

- 保留当前任务的 `session_id`
- 保留当前任务历史、文件与工作区目录
- 仅替换当前活跃的 Kimi runtime / agent 实例

因此产品文案不应写成“热加载成功”，而应写成：

- 已保存，下次执行前会刷新当前任务运行态
- 当前运行态仍在使用旧配置
- 立即应用会刷新当前任务运行态，但不会清空历史

---

## 12. 变量与密钥架构

凡是涉及 token、api key、secret、password、private endpoint 的配置，默认必须拆成：

- 变量
- 密钥

变量可查看和编辑，密钥默认遮蔽，只允许写入、替换、删除，不默认回显原值。

前端“变量与密钥”页管理的是 AIASys 的受控配置层，而不是宿主机真实环境变量列表。

但需要补充一个当前阶段例外：

- 当前产品默认面向单机个人使用
- 在 MCP 配置这一条链路上，允许先不做变量与密钥分离
- token、api key 可以先直接写在工作区或仓库内的 MCP `yaml` 文件中

也就是说，变量与密钥分层是后续可演进方向，不是当前单机个人阶段的强制前置条件。

---

## 13. 深度研究模式在架构中的位置

当前代码里，深度研究仍以 mode/preset 形式存在，但这不应再被当作长期一级架构骨架。

目标态应继续往下收口：

- `数据分析` 与 `深度研究` 都不再长期保留为一级产品模式
- 它们更适合作为统一主控下的内置工具包 / 工作流预置
- 顶层只保留一个默认助手与任务工作区主骨架
- 差异主要落在默认能力包、提示词 overlay、任务配置预置和专业子角色上

也就是说，深度研究既不是独立的大骨架，也不应继续和数据分析一起维持“双模式 + 双主控”心智。

---

## 14. 当前必须避免的架构偏差

当前架构中应避免以下问题：

1. 把 `mode` 写成能力硬限制。
2. 把市场、默认配置、任务覆盖塞进一个页面假装统一。
3. 把“新建工作区”“新建对话”“清空记忆”做成同一个动作。
4. 为同一个概念创建多个平行真相源。
5. 把系统级目录项误写成用户已经启用的配置。
6. 把可保存误写成运行中热更新。
7. 把 MCP 提升成所有能力的唯一总模型。
