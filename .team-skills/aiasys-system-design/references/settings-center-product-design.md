# AIASys 设置中心产品设计

状态: 主线专题
更新时间: 2026-04-04
适用版本: v0.3.9 当前主线

---

## 1. 文档目的

本文档用于收敛 AIASys 当前“设置中心”的产品定位、边界和信息架构。

它回答的是：

- 设置中心在 AIASys 里到底是什么
- 哪些内容应该进入设置中心，哪些不应该进入
- 设置中心与任务工作区、市场、账户域之间如何分工
- 当前前端为什么优先采用弹窗式设置中心，而不是继续扩张独立设置路由页

本文档属于系统架构主稿下的专题文档。若与当前运行中代码不一致，应先核对当前实现与 active task session，再决定是否回写本文件。

---

## 2. 核心定位

### 2.1 设置中心是什么

设置中心是 AIASys 的用户默认配置中心。

它负责承接用户跨任务长期复用的默认配置，而不是承接某个任务这一次的即时工作状态。

设置中心回答的是：

- 我通常希望系统如何工作
- 我的长期默认模型、Agent、偏好、环境和扩展如何配置

而不是：

- 当前这个任务现在怎么跑
- 当前执行这一轮实际用了什么

### 2.2 设置中心不是什么

设置中心不是：

- 任务工作区的替代入口
- 市场目录本身
- 当前任务配置弹层
- 账户资料页的别名
- 所有系统页面的总收纳筐

如果一个对象只对当前任务生效，它应优先留在任务工作区，而不是进入设置中心。

---

## 3. 当前分层关系

### 3.1 设置中心与任务工作区

两者必须分层：

- 任务工作区负责当前任务实际工作状态
- 设置中心负责用户长期默认配置

当前推荐语义是：

- 任务内修改回答“这个任务这次怎么跑”
- 设置中心修改回答“我以后默认想怎么用系统”

因此，任务级 Agent 配置、任务级能力覆盖、任务级运行环境选择，不应回写成设置中心的主内容。

### 3.2 设置中心与市场

设置中心可以承接市场入口，但不应把市场和设置混成一个概念。

正确关系是：

- 市场回答“系统提供什么”
- 设置中心回答“我默认启用了什么”

因此：

- Skill 市场可以作为设置中心下的一个 section
- MCP 市场也可以作为设置中心下的一个 section
- 但市场条目本身仍属于目录层，不属于设置层事实源

### 3.3 设置中心与账户域

账户域用于承接个人资料、身份信息、账号相关操作。

设置中心用于承接系统使用方式相关的长期默认配置。

因此 `/profile` 不应继续混入模型、Agent、环境或扩展配置入口。

---

## 4. 当前推荐的交互模型

### 4.1 统一设置弹窗优先

当前阶段，AIASys 更适合采用统一 `SettingsDialog` 作为用户级设置主入口。

原因是：

- 用户多数修改发生在任务工作流中，不应频繁跳离当前上下文
- 用户默认配置和任务级配置已经分层，用户级配置更适合在同一系统壳内完成
- 独立设置大页容易继续膨胀成第二工作台

因此当前推荐模型是：

- 主入口优先使用统一设置弹窗
- `/settings/*` 可以作为兼容壳、调试入口或深链接承接页
- 但不继续扩张为独立主工作台

### 4.2 设置弹窗内部应采用 section 分层

设置中心内部推荐按长期默认配置语义分 section，而不是按技术实现对象堆页面。

当前合理 section 包括：

- Agent 默认配置
- 模型与偏好
- 运行环境
- Skill 市场
- MCP 市场
- 变量与密钥

后续如有新增，也应优先判断是否属于长期默认层，再决定是否进入设置中心。

---

## 5. 当前应进入设置中心的内容

### 5.1 Agent 默认配置

进入设置中心的应是用户默认 Agent 配置。

它回答的是：

- 新建任务时默认采用什么 Agent 配置
- 用户的长期偏好和默认版本是什么

### 5.2 模型与偏好

模型选择、偏好设置、默认行为参数等长期默认项，适合进入设置中心。

### 5.3 运行环境

设置中心可承接运行环境管理入口，但这里表达的是长期默认层和全局可用环境管理，而不是某任务当前选中的运行环境事实。

### 5.4 Skill 市场与 MCP 市场

`Skill 市场`、`MCP 市场`、已安装扩展、后续集成管理，适合通过设置中心进入。

这里的重点是：

- 设置中心承接入口与默认安装管理
- 不直接替代市场的目录语义

对 Skill 来说，设置中心和市场页后续更适合提供的是：

- 安装包预览
- 已安装看板
- 卸载操作
- 加载到当前工作区的入口

而不是把“编辑单个 `SKILL.md`”作为主入口。

### 5.5 变量与密钥

变量与密钥属于用户长期受控配置层，适合进入设置中心。

但必须坚持：

- 变量与密钥不是宿主机环境变量总览
- 密钥默认遮蔽
- 不把共享 token 当作前端可见公共信息

---

## 6. 当前不应进入设置中心的内容

以下内容不应继续被塞进设置中心主结构：

- 当前任务对话内容
- 当前任务执行记录
- 当前任务级 Agent 配置
- 当前任务级能力覆盖
- 当前任务级运行环境选择
- 任务内文件和资料操作
- 任务内知识与数据库阅读视图

这些内容应继续留在任务工作区。

---

## 7. 路由与承接策略

### 7.1 当前主入口

当前主入口应优先是任务上下文内的设置弹窗。

### 7.2 兼容入口

`/settings`、`/settings/*` 当前可以保留，但应解释为：

- 兼容入口
- 深链接入口
- 调试或开发期承接页

而不是继续发展成与任务工作区并列的第二主工作台。

### 7.3 页面承接原则

如果某个旧设置页仍存在，后续处理顺序应是：

1. 先确认它表达的是长期默认层还是任务层
2. 若属于长期默认层，则并入设置中心 section
3. 若属于任务层，则迁回任务工作区
4. 再决定旧路由是否保留为兼容壳

---

## 8. 面向后续实现的约束

### 8.1 前端约束

- 设置中心优先做 section 收口，不优先继续扩张路由树
- 不把账户页和设置页重新混写
- 不把市场说明、任务说明和内部架构说明直接渲染为用户提示
- 设置项过多时默认支持分组和折叠

### 8.2 后端约束

- 后端可以逐步补设置 overview read model
- 但不要把所有用户配置粗暴揉成单一大对象
- 仍保持 Agent、Preferences、Runtime、Capability 安装状态等服务分层

### 8.3 文档约束

- 设置中心文档必须始终明确它是“长期默认层”
- 若某功能同时有任务级和用户级两个入口，文档必须明确两者作用差异
- 设置中心相关专题不应再次漂移成“系统所有功能汇总页”说明书

---

## 9. 本专题的当前结论

当前 AIASys 的设置中心应理解为：

1. 一个承接用户长期默认配置的统一入口。
2. 主入口优先采用统一设置弹窗，而不是独立第二工作台。
3. 与任务工作区严格分层，不承接任务即时工作状态。
4. 可以承接 Skill 市场、MCP 市场、运行环境和变量密钥等长期配置入口。
5. 与账户域分开，不再把 `/profile` 当作系统配置中心。
6. `/settings/*` 可以保留为兼容壳，但不再继续扩张为主骨架。

这份专题文档用于约束后续设置中心、扩展与集成、变量与密钥、环境管理入口的持续重构。
