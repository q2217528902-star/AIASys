# AIASys 任务工作区信息架构

状态: 主线专题
更新时间: 2026-04-12
适用版本: v0.3.9 当前主线

---

## 1. 文档目的

本文档用于收敛 AIASys 当前“任务工作区”这一核心对象的信息架构。

它回答的是：

- 任务工作区在产品上到底是什么
- 主工作区里应该放什么，不应该放什么
- 左侧、主区、次级入口、设置、市场之间如何分工
- 当前前端重构时，哪些入口可以收口，哪些功能不能直接消失

本文档属于系统架构主稿下的专题文档。若与运行中代码不一致，应先核对当前实现与 active task session，再决定是否回写本文件。

对应的可视化骨架示意可参考：

- `.agents/skill-development/aiasys-system-design/examples/task-workspace-shell-v1.html`

---

## 2. 核心定义

### 2.1 任务工作区是什么

AIASys 当前的一等产品对象是任务工作区。

任务工作区不是“一次性对话”，也不是“项目空间”“研究团队”“多层编排容器”。

它更接近于：

- 一个可长期使用的工作单元
- 一个围绕具体目标持续积累上下文的工作区
- 一个同时承接对话、资料、配置、执行记录和阶段性产出的容器

实现层当前仍大量依赖 `session / conversation` 命名，但在产品语义上统一解释为任务工作区。

### 2.2 工作区下的对话是什么

对话是任务工作区内部的二级对象。

一个任务工作区下应允许存在多个对话，用来承接：

- 不同阶段的记忆轮次
- 同一任务下的不同思路分支
- 清空记忆重新开始后的新入口
- 围绕某个具体问题展开的短线推进

因此当前信息架构要区分三件事：

- 新建任务工作区
- 在当前工作区中新增对话
- 切换当前工作区中的历史对话

当前还应补一类能力：

- 基于当前或历史对话内容 Fork 出新的分支对话

### 2.3 一个任务工作区里长期积累什么

一个任务工作区中，用户会持续积累：

- 对话与历史消息
- 文件与资料
- 执行记录与运行证据
- 当前任务级配置
- 与该任务相关的知识、资源和工具入口

因此任务工作区必须支持长期使用，不应被设计成只适合一次性问答的轻量壳。

## 3. 当前信息架构原则

### 3.1 主骨架只有一个

当前产品主骨架必须围绕任务工作区展开。

这意味着：

- 不再额外并行维护“研究任务骨架”“项目空间骨架”“资源广场骨架”
- 不再让左侧导航同时承担任务导航、系统导航、项目导航三种职责
- 不再把系统级目录页直接塞进任务主工作流里

### 3.2 主区优先承接当前活跃对象，协作默认停靠右侧

任务工作区的主区优先承接当前正在进行的工作。

当前主区的优先内容是：

- 当前活跃对象
- 与当前对象直接相关的即时反馈
- 与当前工作紧耦合的对象级操作入口

不应直接把以下内容长期占据主区：

- 用户级默认设置
- 系统市场目录
- 与当前任务弱相关的全局资源管理页

这里的“当前活跃对象”默认应包括：

- `workspace explorer`
- `notebook`
- `database`
- `pdf`
- `file`
- `chart`
- `artifact`

右侧默认承接协作层：

- 当前分支对话
- 输入区
- 当前对象 / 当前任务摘要
- `Runs / Plan / Resources` 等可展开模块

这条骨架的目标是避免用户在“聊天主画布”和“对象主画布”之间反复切换。

### 3.3 旧入口可以迁移，但不能无去向消失

当收口左侧导航、设置入口或资源入口时，允许改变入口位置，但必须满足：

1. 功能仍然存在
2. 新入口语义更清楚
3. 有明确映射去向

如果只是把入口删掉，却没有新的稳定承接位置，就不算完成收口。

## 4. 任务工作区的页面分工

### 4.1 左侧：任务导航与最小系统入口

左侧导航的职责应保持最小化，优先只承接：

- 新建任务
- 任务搜索
- 工作区列表
- 当前工作区下的对话列表
- 账户与全局入口

当前命名口径建议统一为：

- 一级主动作：`新建任务`
- 工作区内部动作：`新建对话`
- 对话项次级动作：`Fork 对话`

左侧不应继续膨胀为：

- 项目空间导航
- 研究团队导航
- 系统功能广场
- 资源能力大杂烩入口墙

### 4.2 中间主区：当前任务的核心工作面

中间主区是任务工作区的核心工作面，主要承接：

- 当前活跃对象
- 当前对象的主内容与主交互
- 与当前对象直接相关的结果反馈
- 与当前对象紧耦合的局部导航

当前主线下，中间主区优先表达的对象应包括：

- `workspace explorer`
- `notebook`
- `database`
- `pdf`
- `file`
- `chart`
- `artifact`

其中 `workspace explorer` 负责承接工作区目录、文件树、资产入口和对象切换，不再默认把文件树长期塞在右侧与聊天抢空间。

如果当前没有明确活跃对象，中间主区可以停留在：

- 工作区空态
- 对象选择态
- notebook 空白态

但不应因此把聊天重新抬回唯一主画布。

### 4.3 右侧：默认承接 assistant sidecar

右侧默认应承接当前任务的协作层，而不是与中间主区并列出第二个对象主区。

右侧默认应优先展示：

- 当前分支聊天正文
- 输入区
- 当前对象摘要
- 当前任务摘要
- `Runs / Plan / Resources` 等可展开模块

对话记录的正确位置不是单独再做一个需要频繁切换的一级 tab，而是：

- 作为右侧最外沿的历史边条
- 在侧栏宽度足够时自动露出
- 空间不足时收进 `...` 或抽屉

执行记录、日志、数据库阅读视图、知识图谱等内容可以继续存在，但更适合作为从当前对象或 `Task` 模块进入的次级入口，而不是与工作区或聊天并列做成一级常驻主区。

### 4.4 弹窗与对话框：承接任务级配置

任务级配置更适合放在任务工作区上下文内的弹窗或对话框中。

当前统一入口应逐步收口为：

- 任务级 Agent 配置
- 任务级能力配置
- 任务级运行环境配置

它们共同回答的是：

> 这个任务工作区这次怎么跑。

当前主线下，这里的“这次怎么跑”默认回答的是工作区层配置，而不是当前对话单独复制一套完整配置。

## 5. 任务工作区与设置、市场的边界

### 5.1 设置不属于任务主区

设置层负责用户长期默认配置。

因此以下内容不应继续作为任务工作区主区的常驻一部分：

- 用户默认 Agent 配置
- 模型与偏好
- 变量与密钥管理
- 全局运行环境管理

这些内容应通过统一设置入口进入，而不是写回任务工作区主区。

### 5.2 市场不属于当前任务事实层

Skill 市场、MCP 市场等目录页属于供给侧目录。

它们回答的是：

- 系统提供什么
- 用户可以安装什么

而不是：

- 当前任务正在使用什么

因此市场应与任务工作区配置层分开。

### 5.3 任务配置也不承担市场职责

任务配置只负责当前任务覆盖，不负责解释全量目录。

如果任务配置页里出现大量目录型内容，应优先改成：

- 摘要
- 推荐
- 当前任务覆盖结果

而不是把市场整页塞进去。

## 6. 当前推荐的入口模型

### 6.1 一级对象

当前一级对象应收敛为：

- 任务工作区
- 设置
- 市场

### 6.2 二级对象

在任务工作区内，再承接以下二级对象：

- 对话列表
- 当前活跃对象
- 右侧 assistant sidecar
- 对话记录边条
- 对话 Fork / 分支入口
- 任务配置
- 文件与资料
- 工作区摘要
- 资源与能力快捷入口
- 执行记录与证据入口

### 6.3 资源与能力入口的正确定位

资源与能力入口可以存在，但它应是任务工作区内的二级承接层，用来集中承接历史遗留入口。

它不应被继续解释成新的一级主导航，也不应在页面中夹带架构注释式说明。

---

## 7. 路由与实现命名的当前口径

当前前端和后端仍存在大量 `session / conversation / analysis` 命名，这是实现真相的一部分。

当前口径应为：

- 路由与接口可继续沿用现有实现命名
- 新的产品文案优先解释为任务工作区
- 不为了表面统一而立刻全仓重命名

因此：

- `/analysis/{session_id}` 当前仍可视为“当前工作区中的当前对话入口”
- `session status`、`conversation history` 等实现对象仍然有效
- 产品文档和新界面文案应尽量减少把这些实现名词直接暴露成用户主语义

当前实现参考上，可以把 vendored `kimi cli` 的 session fork 能力视为“对话分支”的技术参考，而不是直接照搬它的 session-first 产品模型。

更准确地说：

- `kimi cli` 的 `/fork` 与 web fork 更接近“从现有会话派生新分支”
- AIASys 后续应把这类能力解释为“当前工作区内的新对话分支”
- 它不应被解释为“新建任务工作区”

---

## 8. 当前不再继续推进的方向

当前阶段不应把以下对象继续推进为用户主骨架：

- `ResearchProject`
- `ProjectLeadSession`
- `ResearchTeam`
- 独立的“研究任务”导航骨架
- 与任务并行的项目空间主入口

这些材料若保留，只能作为历史探索或候选方案归档，不再作为当前主线默认入口理解。

---

## 9. 面向后续实现的约束

### 9.1 前端约束

- 左侧导航优先继续做减法
- 左侧优先做到“工作区为主、对话为辅”，不要把系统入口重新膨胀回去
- 任务主区保持工作面纯度
- 任务配置在当前上下文中完成，不强制跳独立大页
- 资源与能力入口只承接入口映射，不承接解释层膨胀
- 不把内部设计说明直接渲染给用户

### 9.2 后端约束

- 继续允许底层沿用 `session` 作为事实源
- 对外逐步补 `workspace / conversation / run` 相关 read model / projection
- 不把任务工作区语义硬塞成新的底层核心表结构改名任务

### 9.3 文档约束

- 当前主线文档应继续以“任务工作区”为主对象
- 若某功能仍保留旧入口，必须说明当前去向
- 过时探索材料应物理归档，不只做“已过时”标注

---

## 10. 本专题的当前结论

当前 AIASys 的信息架构主线应理解为：

1. 任务工作区是唯一主骨架。
2. 单个任务工作区下允许存在多个对话，对话是工作区内部的二级对象。
3. 左侧以工作区导航为主，并承接当前工作区下的对话切换，不再并行承载项目骨架和系统广场。
4. 中间主区承接当前工作区中的当前对话工作面。
5. 右侧默认只承接当前任务工作区，执行证据和诊断信息通过次级入口展开。
6. 任务级配置在当前上下文内统一收口，并默认挂在工作区层。
7. 设置承接用户默认配置，市场承接供给目录，二者都不回灌成任务主区。
8. “新建任务”“新建对话”“清空记忆重新开始”“Fork 对话”必须做成语义不同的动作。

这份专题文档用于约束后续左侧导航、任务配置、资源入口和工作区摘要的持续重构。
