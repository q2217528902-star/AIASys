# AIASys 市场与能力接入模型

状态: 主线专题
更新时间: 2026-04-04
适用版本: v0.3.9 当前主线

---

## 1. 文档目的

本文档用于收敛 AIASys 当前“市场层”和“能力接入层”的产品与架构口径。

它回答的是：

- Skill 市场和 MCP 市场到底是什么关系
- 为什么市场不等于当前启用状态
- 能力为什么要按接入方式建模，而不是统一都叫 MCP
- capability registry 在当前系统里承担什么角色
- `mode`、系统默认、用户默认、任务覆盖之间的关系是什么

本文档属于系统架构主稿下的专题文档。若与当前运行中代码不一致，应先核对当前实现与 active task session，再决定是否回写本文件。

---

## 2. 核心结论

### 2.1 市场是目录层，不是运行态

AIASys 里的市场层，首先表达的是系统供给目录。

它回答的是：

- 系统提供了哪些能力条目
- 这些能力通过什么方式接入
- 用户可以安装、配置或启用哪些条目

它不直接回答：

- 当前任务正在使用什么
- 当前这一轮实际调用了什么
- 当前用户已经永久默认启用了什么

### 2.2 能力接入是总模型，MCP 只是其中一种

AIASys 当前不把 MCP 视为总抽象。

更稳定的总模型是：

- 能力接入

在这个总模型下，再区分具体适配器类型：

- CLI / 本地命令
- 原生 API / SDK
- MCP

因此：

- MCP 市场是协议型目录
- Skill 市场是系统提供的行为包目录
- capability registry 是统一描述层
- 不能把所有外部能力都强行先包成 MCP 才允许进入系统

---

## 3. 市场层的对象分工

### 3.1 Skill 市场

Skill 市场用于展示系统提供的 Skill 包。

它主要承接：

- Skill 名称
- Skill 简介
- 是否已安装或可用
- 可进入的安装或管理操作

Skill 市场更接近“行为包目录”，而不是底层工具协议目录。

当前 Skill 的正确对象应是：

- skill 目录包

而不是：

- 单个 `SKILL.md` 文档

`SKILL.md` 只是目录包入口文件。一个完整 skill 包后续可以包含：

- `SKILL.md`
- 附属 Markdown 文档
- 模板
- 脚本
- 资源文件

因此后续市场和管理面板应围绕“目录包安装与挂载”设计，而不是围绕“单个 Markdown 在线编辑”设计。

### 3.2 MCP 市场

MCP 市场用于展示系统支持的 MCP 接入条目。

它主要承接：

- MCP 名称
- MCP 简介
- 所需配置
- 是否已安装
- 是否可在任务中启用

MCP 市场是协议型目录，不应被解释为“全能力市场”。

当前阶段需要补充一个前提：

- AIASys 当前默认面向单机个人使用
- MCP 优先是“文件驱动的可加载配置”，不是“必须先有可视化编辑器的产品模块”

因此当前对 MCP 的更稳口径是：

- 系统可以提供一份默认 `yaml` 配置模板
- 工作区或仓库里可以放自己的 `mcp.yaml`
- 后端负责解析 `yaml` 并在运行时加载
- 前端至多提供预览和测试，不必把编辑能力做成第一优先级

这意味着当前阶段“有没有前端编辑面板”不是核心问题，核心问题是：

- 配置文件是否清晰
- 加载链路是否稳定
- 报错和测试是否可观察

### 3.3 后续更多目录项

如果后续 AIASys 接入更多能力目录，例如：

- 原生 API 型能力目录
- CLI 型能力目录
- 运行时 helper 目录

也应继续按目录层扩展，而不是硬塞回 MCP 市场。

### 3.4 Skill 安装形态

Skill 的安装形态后续宜支持：

- 系统内置目录
- 工作区本地 skill 目录
- 压缩包导入安装

当前最稳的统一口径是：

- 市场展示的是 skill 包目录项
- 安装落地的是 skill 目录包
- 工作区加载的是显式启用的 skill 包
- `SKILL.md` 只承担入口说明，不承担完整包的全部含义
- 运行时 builtin Skill 目录应独立于协作/开发 skill 仓库，例如 `apps/backend/builtin_skills/`
- `.agents/skills/` 负责 AI 开发协作，不直接冒充最终用户可安装的 runtime Skill 系统目录

## 4. 能力接入模型

### 4.1 三类主适配器

当前能力接入最少应承认三类：

#### 4.1.1 CLI / 本地命令

适用于已有成熟命令行工具链的能力。

特点：

- 执行路径直接
- 更适合本地或容器环境中的现有工具
- 往往与 runtime substrate 关系更近

#### 4.1.2 原生 API / SDK

适用于官方已经提供稳定 HTTP API 或 SDK 的能力。

特点：

- 集成路径通常更直接
- 认证、速率限制和请求结构更清楚
- 不需要为了统一而额外包一层 MCP

#### 4.1.3 MCP

适用于已有 MCP server 生态、需要标准工具协议封装、或需要在 Agent 之间复用同一工具暴露层的场景。

特点：

- 适合协议统一
- 适合生态复用
- 适合以工具调用方式暴露能力

在 AIASys 当前单机个人模式下，MCP 的推荐接入形态可以进一步收口为：

- 以 `yaml` 文件声明 server 列表
- 允许在文件里直接写 token、api key、headers 等认证字段
- 由后端统一转成真实运行时配置

也就是说，当前不需要为了“密钥安全”先把这套个人使用链路设计得过重。后续如果切到团队共享或多人部署，再把认证字段迁移到独立受控配置层。

### 4.2 为什么不能把 MCP 做成总模型

如果把 MCP 直接抬成总模型，会带来几个问题：

1. 原生 API / SDK 能力被迫多绕一层
2. CLI 能力与 runtime 关系被错误扁平化成“工具开关”
3. 市场层会误导用户认为“系统所有能力都属于 MCP”
4. capability registry 会失去对接入方式的真实描述能力

因此 MCP 应保持为重要接入方式，但不是唯一总抽象。

## 5. capability registry 的角色

### 5.1 registry 是统一描述层

capability registry 当前的正确定位是统一描述层，而不是统一执行器。

它负责回答：

- 系统当前有哪些能力条目
- 每个条目属于哪种 kind
- 每个条目通过什么 adapter 接入
- 每个条目默认推荐给哪些 mode
- 每个条目当前是否已安装、可用、激活

### 5.2 registry 当前不负责什么

当前阶段，registry 不直接负责：

- 接管所有执行路径
- 取代运行时服务
- 取代任务级实际覆盖记录
- 取代执行证据层

它是前端和系统文档所依赖的“统一描述与目录投影层”。

### 5.3 registry 推荐最小字段模型

当前可稳定收敛为以下思路：

- capability_id
- display_name
- kind
- adapter_type
- provider
- default_enabled 或 auto_attach_modes
- config_schema
- secret_requirements
- status
- evidence_level

这里的重点是描述层清楚，而不是一次性把所有后端执行逻辑都揉进去。

## 6. 市场、默认层、任务层之间的关系

### 6.1 系统目录层

回答：系统支持什么。

典型对象：

- Skill 市场条目
- MCP 市场条目
- capability registry 中的系统目录项

### 6.2 用户默认层

回答：用户长期默认启用了什么。

典型对象：

- 已安装 Skill
- 我的 MCP
- 默认启用的能力包
- 用户长期默认配置项

但在当前单机个人模式下，MCP 不一定需要先单独做一层“用户默认面板”。

更简单的实现是：

- 仓库或工作区里存在默认 `yaml` 配置文件
- 当前任务直接在这份文件或它的工作区副本上运行

只有当后续真的出现多套长期默认切换需求时，再把这一层显式抬出来。

### 6.3 任务工作层

回答：当前任务这次实际启用了什么。

典型对象：

- 任务级能力覆盖
- 任务级 MCP 覆盖
- 任务级运行环境与 Agent 组合

### 6.4 证据层

回答：这一轮实际用了什么。

典型对象：

- 执行记录
- 调用证据
- 实际使用的能力快照
- 运行时状态与结果

## 7. mode 与能力推荐的关系

### 7.1 mode 不是能力上限

当前 `mode` 只应表达：

- 默认推荐
- 默认能力包
- 默认自动挂载

而不应表达：

- 硬性能力上限
- 工具白名单封禁
- 绝对不可用边界

### 7.2 mode 的正确作用

例如：

- `research mode` 可以默认推荐搜索、阅读、翻译、记录整理相关能力
- 但不表示其他能力天然不可被任务覆盖启用

因此 mode 更接近“默认预置语义”，不是“运行时封禁规则”。

### 7.3 auto attach 与 hard limit 的区别

当前更稳定的字段语义应是：

- `auto_attach_modes`

它表达的是：

- 哪些 mode 下默认自动挂载该能力

它不表达：

- 只有这些 mode 才允许使用该能力

## 8. 当前 Skill / MCP 应用机制

### 8.1 不承诺运行中热加载

基于当前 Kimi 底层 runtime 的工作方式：

- Skill 目录在 runtime 创建阶段发现并索引
- MCP 配置在 agent 创建阶段装载

因此当前系统不应承诺：

- 运行中热加载 Skill
- 运行中热替换 MCP

### 8.2 推荐机制：工作区可编辑 + 运行态软重建

当前更合适的机制是：

1. 用户在工作区中修改 Skill 包或 `mcp.yaml`
2. 系统先保存这些改动
3. 当前任务被标记为 `runtime_refresh_required`
4. 当前 turn 结束前不强制中断
5. 下一次执行前，系统重建当前任务的 runtime
6. 新 runtime 读取最新的 Skill / MCP 配置

### 8.3 为什么当前要这样设计

这样做的好处是：

- 符合底层 Kimi runtime 的真实能力边界
- 不需要新建任务
- 不需要清空历史上下文
- 对用户来说仍然是“我改完配置，下一轮就生效”

因此当前最准确的产品语义不是“热更新”，而是“刷新当前任务运行态”。

## 9. 系统级、用户级、任务级的关系

### 9.1 系统级

系统级表示：系统提供该能力目录项。

不表示：

- 全员强制启用
- 管理员代所有用户安装完成

### 9.2 用户级

用户级表示：用户把系统提供的目录项安装或纳入自己的默认配置。

例如：

- 把某个 MCP 安装到“我的 MCP”
- 将某个 Skill 纳入自己的默认使用集合

### 9.3 任务级

任务级表示：当前任务工作区这次实际采用哪些能力覆盖。

它应优先发生在任务配置层，而不是市场层。

## 10. 页面和接口上的推荐落点

### 10.1 市场页面该展示什么

市场页面应优先展示：

- 条目名称
- 一句话简介
- adapter / kind 标签
- 当前状态
- 安装或配置入口

如果是 Skill 条目，建议增加：

- 包内容预览入口
- 已安装状态
- 卸载入口
- 加载到当前工作区入口

不要展示过多内部实现说明，也不要把市场页做成任务配置页。

### 10.2 设置中心该承接什么

设置中心可承接：

- 已安装能力管理
- 默认启用管理
- 与市场相连的入口
- 配置密钥与变量入口

但设置中心不应替代市场目录本身。

### 10.3 任务配置该承接什么

任务配置应承接：

- 当前任务能力覆盖
- 当前任务是否启用某能力
- 当前任务与默认推荐之间的差异

而不是展示整个市场总目录。

对 Skill / MCP 这类会触发 runtime 重建的对象，还应额外展示：

- 当前运行态正在使用的版本
- 已保存但待应用的版本
- 是否需要刷新当前任务运行态

## 12. 本专题的当前结论

当前 AIASys 的市场与能力接入模型应理解为：

1. 市场首先是目录层，不是运行态。
2. 能力接入是总模型，MCP 只是重要接入方式之一。
3. Skill 市场与 MCP 市场并列存在，但承接不同类型的目录语义。
4. capability registry 是统一描述层，不是统一执行器。
5. `mode` 只负责默认推荐与自动挂载，不负责能力硬限制。
6. 系统级、用户级、任务级、证据级必须继续分层，不得重新混写。
7. Skill 与 MCP 当前都不做运行中热加载，而通过“刷新当前任务运行态”生效。

这份专题文档用于约束后续 MCP 市场、Skill 市场、capability registry、默认能力包与任务能力覆盖的继续实现。
