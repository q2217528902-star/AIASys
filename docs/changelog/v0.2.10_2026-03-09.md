# Changelog v0.2.10 (2026-03-09)

## 版本判断

按当前连续的正式版本线，本次更新归并为 `v0.2.10`，而不是额外跳到新的主线编号：

- 变更重点是模型接入、默认模型切换、Agent 兼容性修复和验证链路增强
- 对现有产品形态是增强和纠偏，不是一次新的大功能域切换
- 虽然影响面较大，但整体仍属于当前连续版本线内的稳定性与可用性提升

## 新增能力

### 接入 OpenAI Responses 兼容模型服务
- 系统新增第三方 `openai_responses` provider，接入 `http://47.105.122.4:8317/v1`
- 新增模型 `openai-responses-8317-default`，模型名配置为 `gpt-5.4`
- 已确认该服务支持标准 Responses API 交互，且 `/models` 可正常返回模型列表

## 默认模型策略调整

### 系统默认模型切换到 `gpt-5.4`
- 系统默认 provider 已切换为 `openai-responses-8317`
- 系统默认 model 已切换为 `openai-responses-8317-default`
- `dashscope` 不再作为系统默认，仅保留为个人测试用途

### Agent 默认选模逻辑收敛
- `agent_service` 现在会优先尊重系统默认模型，只要该默认模型属于适合 Agent 路径的 `kimi` 或 `openai_responses`
- 这使得前端和后端在“不显式传 model_id”时，能够一致地命中新的系统默认模型

## 兼容性修复

### 修复 Responses 代理对 reasoning level 的严格要求
- 实测发现该代理在 Responses 请求中不接受空 reasoning level，会返回：
  - `level "" not supported, valid levels: low, medium, high, xhigh`
- 为避免大面积改动 vendor 逻辑，当前通过模型能力配置将该模型标记为 `always_thinking`
- 这样可复用现有 `kimi-cli` 选模与 thinking 逻辑，稳定为该代理携带 reasoning level

### 修复运行环境与会话生命周期串台

#### 问题描述
- 现象：
  - 对话进行中创建新任务时，前台会话可能被过早切走，导致消息区看起来“被吞”
  - 当前会话真实沙盒模式会被“下次新建任务默认模式”污染
  - Local 会话仍可能走到 Docker 预热语义，导致长时间显示“创建中”
- 原因：
  - 新建任务缺少明确的 `prepare -> bind -> activate` 两阶段生命周期
  - 前端将默认沙盒模式和当前会话真实模式混用
  - 后端 `active` / `prewarm` 对会话级 `sandbox_mode` 的真实口径回传不足

#### 改动内容
- 前端引入统一 `newTaskStage` 状态机，收口新建任务和管理面板创建任务流程
- 增加 `currentSandboxMode`，确保当前会话切环境读取会话真实模式
- 删除当前活跃会话时，前端先切到新的空白草稿，再清理旧会话和后端资源
- `GET /api/runtime-envs/sessions/{session_id}/active` 返回 `sandbox_mode`
- `POST /api/agent/prewarm` 优先读取会话元数据，Local 模式直接 no-op success
- `POST /api/runtime-envs/sessions/{session_id}/switch` 的请求模型补齐 `sandbox_mode` 字段，消除本地沙盒切换回归
- 增加后端定向测试，覆盖 Local 切环境、`active` 返回模式、已有 Local 会话 prewarm 语义

#### 文件变更
- `apps/web/src/pages/DataAnalysisPage/hooks/useRuntimeEnvironmentPanel.ts`: 新建任务/切环境状态机与当前模式解析
- `apps/web/src/pages/DataAnalysisPage/hooks/useCodeExecutor/useSessionOrchestrator.ts`: 删除活跃会话时的 replacement draft 生命周期
