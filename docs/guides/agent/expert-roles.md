# 协作专家

> 当前版本: v0.3.9

本文档说明协作专家（Expert Roles）的概念、配置和使用方式。

## 什么是协作专家

协作专家是可选的子 Agent 角色，每个专家有独立的提示词和工具权限。主控 Agent 在需要时将子任务派发给合适的专家执行。

协作专家不是独立的 Agent 实例，而是被主控 Agent 调用的子执行单元。主控 Agent 负责拆解任务、选择专家、派发指令、汇总结果。

## 专家市场

系统提供以下协作专家候选：

| 专家 | 角色 | 适用场景 |
|------|------|----------|
| coder（代码专家） | 主控明确的任务指派范围内的代码实现与调试 | 需要修改普通文件、调试错误、运行构建或测试时 |
| researcher（研究员） | 调研与分析 | 需要搜索资料、阅读论文、整理调研报告时 |
| reviewer（审查员） | 代码与方案审查 | 需要审查代码质量、检查方案合理性时 |

## 两级启用

协作专家的启用分为两级：

### 我的默认

启用到"我的默认"后，该专家在所有工作区中可用。适合常用的通用专家（如 coder、worker）。"我的默认"配置存储在用户全局工作区中。

### 当前工作区

启用到"当前工作区"后，该专家仅在当前工作区可用。适合特定项目需要的专家组合。工作区级启用覆盖"我的默认"——如果某个专家在工作区中禁用，即使"我的默认"中已启用也不会生效。

## 工具策略

每个协作专家有自己的工具上限和执行策略：

- **提示词**：定义专家的角色、能力边界和行为规范
- **工具列表**：定义专家可以调用哪些工具
- **工具上限**：工作区可以进一步裁剪专家的工具子集，只保留当前任务需要的工具

工作区裁剪规则：只能从专家原有工具列表中减少，不能新增专家原本没有的工具。

内置角色的默认边界：

- `coder` 可以在主控明确的任务指派范围内读写普通文件、执行命令并运行验证；不创建、编辑或运行 notebook，只回看已有 notebook 输出。
- `researcher` 和 `reviewer` 默认只读，可以读取文件、notebook 摘要、知识库和图谱证据，也可以搜索或加载 Skill 指南；不写文件、不执行代码、不修改环境变量、不启用或禁用 Skill。

## 执行树

当主控 Agent 派发子 Agent 时，右侧侧栏的执行流会展示执行树结构：

```
主控 Agent（根节点）
├── coder #1（运行中）
├── researcher #1（已完成）
│   ├── 工具: search_web
│   └── 工具: read_file
└── worker #1（失败）
    └── 工具: shell_exec
```

节点颜色：

- **蓝色**：运行中
- **绿色**：已完成
- **红色**：执行失败

点击节点可查看详细信息：

- 对话历史：该子 Agent 的完整执行过程
- 工具调用记录：每个工具调用的输入参数和返回结果
- 产出文件：该子 Agent 在工作区中生成的文件列表

## 子 Agent 上下文隔离

子 Agent 之间的上下文严格隔离：

- Agent A 的数据不会污染 Agent B 的记忆
- Agent B 生成的文件不会覆盖 Agent A 的输出
- 每个子 Agent 有独立的会话上下文，互不可见

主控 Agent 通过派发指令将必要信息传递给子 Agent，子 Agent 完成后将结果返回主控 Agent。主控 Agent 决定哪些信息需要在子 Agent 之间共享。

## 访问入口

### 执行树

点击左侧 Activity Bar 的"专家协作节点"图标，打开执行树面板。面板以树状结构展示当前会话中所有子 Agent 的执行状态。

执行树面板支持：

- 折叠/展开节点
- 点击节点查看详情
- 按状态筛选（全部/运行中/已完成/失败）

### 专家市场

点击左侧 Activity Bar 的"能力管理"图标，在面板中选择"协作专家"分类。面板分为三个区域：

- **专家市场**：浏览所有可用专家，查看每个专家的描述和工具列表
- **我的默认**：管理跨工作区启用的专家
- **当前工作区**：管理当前工作区的专家启用和工具裁剪

## spawn 权限

子 Agent 的派发由主控 Agent 的 `allow_subagent_spawn` 标志控制。该标志在各层配置中设置，优先级规则与三层配置一致。

如果 `allow_subagent_spawn` 为 false，主控 Agent 不会派发任何子 Agent，所有任务由主控自己完成。
