# 上下文压缩

复杂任务中 Agent 的工具调用和中间结果会迅速膨胀上下文，接近模型上下文窗口上限。上下文压缩在保护关键信息的前提下削减上下文体积。

## 两层压缩

### 第一层：无损截断（Snip）

对过长的工具输出进行截断：

- 保留前 50% 和末尾 25% 的内容
- 中间部分标记为已截断，保留行数和字节数统计
- 不改变保留内容的格式和语义

这一层是纯规则截断，不涉及 LLM，速度快且确定性高。

### 第二层：LLM 结构化摘要

调用 LLM 将消息列表压缩为结构化 XML。摘要按 6 个优先级组织信息：

1. 当前焦点：正在处理的任务和当前状态
2. 已完成任务：已完成的步骤和结果
3. 活跃问题：未解决的错误和阻塞
4. 代码状态：修改了哪些文件、当前代码的关键状态
5. 环境信息：工作区路径、运行中的进程等
6. 其他上下文

压缩后的 XML 摘要体积远小于原始消息列表，但保留了继续推进任务所需的核心信息。

## 保护机制

**系统提示词不压缩**：Agent 行为约束相关的系统提示词始终完整保留，不受压缩影响。

**工具调用对保护**：tool_call 和 tool_result 作为边界保留，不会被拆分到不同压缩块中。压缩粒度是消息边界，不是 token 边界。

**多模态内容剥离**：压缩后的消息中移除图片 base64 数据，只保留文本内容和文件引用。压缩后的 Agent 不再能直接"看"到之前的图片，但保留了文件路径可以重新读取。

## 优雅降级

LLM 摘要如果失败（最多重试 3 次，指数退避），返回原始消息列表，不丢失任何状态。压缩失败不会导致任务中断。

## 手动压缩

用户可以从对话栏头部（TokenUsageBar）手动触发上下文压缩。点击后按钮会进入“压缩中”状态，显示进度提示；压缩完成后会刷新上下文占用数据。手动压缩通常用于以下场景：

- 对话历史很长，但还想在当前会话继续
- TokenUsageBar 显示上下文占用接近窗口上限
- 切换话题前清理旧上下文

## 上下文占用显示

TokenUsageBar 在对话栏头部显示：

- 当前上下文占用 token 数
- 模型上下文窗口大小
- 占用比例

这是当前轮提示词大约占用的窗口比例，不是会话累计消耗。

## 常见问题

**Q: 上下文压缩消耗的 token 会计入会话预算吗？**

不会。压缩是系统为维护上下文健康而自动触发的机制，不属于用户主动发起的对话或 AutoTask 行为。会话预算（TokenUsageBar 中显示的 `tokens_used`）只统计主对话 LLM 调用的输入输出 token，不统计压缩模型产生的消耗。

这避免了"上下文超限需要压缩 → 但预算已耗尽无法压缩 → 上下文继续膨胀"的死锁。压缩模型支持独立配置（`task_models.compaction`），建议使用廉价模型进一步控制成本。

**Q: 压缩后为什么上下文占用没有明显下降？**

可能的原因：

1. **保留消息本身很长**：压缩只替换被 summary 覆盖的旧消息，最近 N 条 user/assistant 消息会原样保留。如果保留的 assistant 回复包含了大量代码或数据，体积依然可观
2. **System prompt 很大**：系统提示词不参与压缩，如果注入了较多 skill 描述或记忆内容，这部分固定占用始终存在
3. **刚刚压缩完又新增了多轮对话**：压缩触发后，如果继续多轮交互，上下文会重新增长

**Q: 压缩会丢失信息吗？**

压缩的本质是信息有损精简。LLM 摘要会保留关键结论和当前状态，但可能丢失：

- 中间尝试过程的细节（如第 3 版代码为什么不行）
- 过长的工具输出内容（已被 snip 截断）
- 图片内容（被替换为文件路径引用）

如果担心信息丢失，可以在压缩前手动保存关键结论到工作区文件，或让 Agent 将重要发现写入 `MEMORY.md`。
